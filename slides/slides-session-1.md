## 2. The “What” and “Why” of Open Source

**Open Source** means that the source code is freely available to use, modify, and share.

### Why people open source:
- Collaboration and learning.
- Faster innovation and iteration.
- Building reputation and community.
- Ensuring transparency and longevity.

---

## 3. Open Source vs Free Software

| Aspect | Open Source | Free Software |
|---------|--------------|----------------|
| Focus | Practical benefits | Ethical freedom |
| Goal | Collaboration, quality, reuse | Freedom to use and share |
| Example | GitHub projects, permissive licenses | GNU tools, FSF philosophy |

Both movements value openness — they just emphasize different aspects.

---

## 4. Git & GitHub Basics

### Git — the version control tool
- Tracks changes in files over time.
- Lets multiple people collaborate safely.

### GitHub — the collaboration platform
- Hosts repositories (projects).
- Provides tools for issues, pull requests, discussions, and automation.

---

## 5. Open Source Lingo — Glossary

| Term | Meaning |
|------|----------|
| **Repo** | A project containing files and history. |
| **Fork** | Your personal copy of someone else’s repo. |
| **Clone** | Download a repo locally. |
| **Commit** | Save a change with a message. |

---

| Term | Meaning |
|------|----------|
| **Branch** | A parallel version of a repo. |
| **PR (Pull Request)** | A proposal to merge your changes. |
| **Issue** | A conversation for bugs or features. |
| **Maintainer** | Person managing the repo. |
| **Contributor** | Anyone submitting improvements. |

---

## 6. Anatomy of a Repository

Typical structure:

```
my-project/
├── README.md           # Overview and usage
├── LICENSE             # Legal terms of use
├── CONTRIBUTING.md     # How to get involved
├── CODE_OF_CONDUCT.md  # Community expectations
├── .github/            # Templates and workflows
└── src/ or docs/       # Source code or documentation
```

Each file helps maintainers and contributors work smoothly.

## 7. Non-Code Contributions

You don’t have to be a programmer to contribute!

Examples:
- Fixing typos or improving documentation.
- Translating text.
- Designing logos or graphics.
- Testing and reporting bugs.
- Triaging issues or reviewing pull requests.

All these help projects grow and stay healthy.

---

## 8. Maintainer Perspective

Maintainers appreciate contributions that are:

- Clear: easy to understand and review.
- Focused: address one change at a time.
- Documented: include a descriptive commit message.
- Respectful: follow the Code of Conduct.

> Tip: Read the CONTRIBUTING.md before opening a PR — it shows how to help effectively.

---

## 9. Finding Beginner-Friendly Projects

Look for labels such as:
- `good first issue`
- `help wanted`
- `documentation`

Use GitHub filters or sites like [goodfirstissue.dev](https://goodfirstissue.dev) to find entry-level tasks.

Projects with active maintainers and recent updates are best for first contributions.
